# ChronoFlow Exec 技术方案

## 1. 文档定位

本文档用于指导 AI 或开发者在 `chronoFlow-exec` 中实现 ChronoFlow V1 执行器后端。

本文档是开发执行方案，不是概念介绍。实现时以以下文件为需求来源：

1. `../docs/prd-v1.md`
2. 当前项目的 `AI_CODING_RULES.md`
3. 当前项目的 `README.md`
4. 根目录 `docs/TESTING_GUIDE.md`

`chronoFlow-exec` 是部署在 Linux 执行服务器上的轻量执行器，只负责接收 admin 下发的任务、执行 Glue Shell、收集日志、处理超时、处理用户终止、保存 pending callback 文件并回调 admin。

## 2. 最重要的模板偏差

当前后端模板默认是 `Kratos + GORM + MySQL`，并且已经生成了：

1. `configs/config.yaml` 中的 `data.database`
2. `internal/conf/conf.proto` 中的 `Data.Database`
3. `internal/data/data.go` 中的 GORM 初始化
4. `internal/data/model.go`
5. `internal/data/user.go`
6. `api/user/v1/user.proto`
7. `internal/service/user.go`
8. `internal/biz/user.go`

这些内容是模板示例，不是 ChronoFlow Exec 的业务代码。

ChronoFlow V1 明确要求：

```text
chronoFlow-exec 不连接 MySQL。
chronoFlow-exec 不直接读写 admin 的数据库。
chronoFlow-exec 只通过 HTTP 与 chronoFlow-admin 通信。
```

因此执行器实现时必须处理模板默认数据库链路：

1. 移除或禁用 `configs/config*.yaml` 中的数据库配置。
2. 移除或禁用 `internal/conf/conf.proto` 中的 `Data.Database` 和 `Data.Redis`，除非保留但完全不注入、不使用。
3. 移除 `internal/data/data.go` 中的 `NewDB`、GORM 连接、AutoMigrate、事务能力。
4. 不新增任何 DAO、Repository、Migration、GORM Model。
5. 不为了复用模板示例 CRUD 而引入数据库。
6. `go.mod` 中如果没有其他用途，后续实现阶段可以移除 `gorm` 和 `mysql driver` 依赖。

执行器本地持久化只允许使用文件系统：

```text
/app/data/callbacks/pending
/app/data/callbacks/expired
/app/data/tmp
```

## 3. 模板示例代码使用原则

模板自带 `user` 示例模块只用于学习写法。

允许参考：

1. proto 写法。
2. service/biz 分层。
3. 统一错误返回。
4. `ProviderSet` 和 `wire` 注入方式。
5. `internal/server/http.go` 路由注册方式。
6. 测试写法。

禁止：

1. 把 `User` 当作执行器业务模型。
2. 保留无关 `user` 示例接口作为正式业务接口。
3. 继续使用 `internal/data/user.go` 做执行器业务。
4. 为执行器新增数据库 CRUD。

一句话：模板代码是写法样板，不是需求来源。

## 4. 架构边界

### 4.1 Exec 负责

1. 提供 `GET /health`。
2. 提供 `POST /run`。
3. 提供 `POST /kill`。
4. 校验 `X-Executor-Token`。
5. 接收 admin 下发的 Glue Shell。
6. 生成临时 `.sh` 文件。
7. 使用 `/bin/bash` 执行脚本。
8. 为每次任务创建独立 Linux 进程组。
9. 收集 stdout、stderr、exit_code。
10. 控制日志最大 5MB。
11. 处理任务超时。
12. 处理用户终止。
13. 将执行结果写入 pending callback 文件。
14. 调用 admin callback 接口。
15. callback 失败后定期重试。
16. 执行器重启后恢复 pending callback 重试。

### 4.2 Exec 不负责

1. 不连接 MySQL。
2. 不创建数据库表。
3. 不保存最终执行日志。
4. 不管理任务配置。
5. 不解析 Cron。
6. 不判断任务是否应该被调度。
7. 不保存 Glue 到数据库。
8. 不同步 Python 脚本。
9. 不创建 Python 虚拟环境。

## 5. 目录规划

建议在当前模板基础上调整为：

```text
api/
└── executor/v1/executor.proto

internal/
├── auth/
│   └── executor_token.go
├── callback/
│   ├── client.go
│   └── worker.go
├── config/
│   └── validate.go
├── errors/
│   └── codes.go
├── process/
│   ├── manager.go
│   ├── runner.go
│   ├── killer_linux.go
│   └── log_buffer.go
├── store/
│   └── pending_store.go
├── service/
│   ├── executor.go
│   └── service.go
└── server/
    └── http.go
```

说明：

1. `auth` 只处理执行器 token 校验。
2. `process` 处理任务运行、进程组、超时、kill、日志收集。
3. `store` 处理 pending callback 文件。
4. `callback` 处理回调 admin 和重试 worker。
5. `service` 负责 HTTP 入参校验和调用业务模块。
6. 不需要 `internal/data` 数据库层。若保留该目录，只能存放文件型 store，不得出现 GORM。

## 6. 配置设计

需要重写执行器配置，删除数据库配置。

建议 `configs/config.yaml`：

```yaml
server:
  http:
    network: tcp
    addr: 0.0.0.0:10004
    timeout: 30s

executor:
  name: "executor-01"
  token: "${EXECUTOR_TOKEN:change-me}"
  data_dir: "/app/data"
  shell_path: "/bin/bash"
  temp_dir: "/app/data/tmp"
  kill_grace_seconds: 5
  max_log_bytes: 5242880

callback:
  retry_interval_seconds: 30
  pending_retention_days: 7
  request_timeout_seconds: 10

logging:
  level: INFO
  file:
    path: ./
    name: "chronoFlow-exec.log"
    maxSize: 10
    maxBackups: 5
    maxAge: 30
    compress: false
```

配置要求：

1. `executor.token` 生产环境不能为空。
2. `executor.shell_path` V1 固定推荐 `/bin/bash`。
3. `executor.data_dir` Docker 部署时必须挂载。
4. `executor.temp_dir` 用于临时 `.sh` 文件。
5. `callback.pending_retention_days` 默认 7 天。
6. `callback.retry_interval_seconds` 默认 30 秒。
7. `max_log_bytes` 默认 5MB。

修改 `conf.proto` 后运行：

```bash
make config
make wire
```

## 7. 接口设计

### 7.1 鉴权规则

所有执行器接口都必须校验：

```text
X-Executor-Token: <executor_token>
```

错误规则：

1. Header 缺失，返回认证错误。
2. token 不匹配，返回认证错误。
3. token 比较建议使用常量时间比较。

### 7.2 健康检查

接口：

```text
GET /health
```

返回：

```json
{
  "code": 0,
  "message": "Health success",
  "data": {
    "status": "online",
    "executor_name": "executor-01"
  }
}
```

要求：

1. 必须校验 `X-Executor-Token`。
2. 不检查数据库，因为执行器没有数据库。
3. 可以检查 data_dir 是否可访问。
4. 可以返回执行器名称。

### 7.3 执行任务

接口：

```text
POST /run
```

请求：

```json
{
  "job_id": 1001,
  "log_id": 2001,
  "script": "#!/bin/bash\necho hello",
  "timeout_seconds": 600,
  "callback_url": "http://10.0.0.10:8080/internal/job-runs/callback",
  "callback_token": "callback-token-from-admin"
}
```

响应：

```json
{
  "code": 0,
  "message": "Run success",
  "data": {
    "log_id": 2001,
    "status": "accepted"
  }
}
```

处理规则：

1. 校验 token。
2. 校验 `job_id > 0`。
3. 校验 `log_id > 0`。
4. 校验 script 非空。
5. 校验 `timeout_seconds > 0`。
6. 校验 callback_url 非空。
7. 校验 callback_token 非空。
8. 检查同一 `job_id` 是否已经在本执行器运行。
9. 如果同一 `job_id` 正在运行，拒绝请求。
10. 创建运行实例并立即返回 accepted。
11. 后台 goroutine 执行脚本。

### 7.4 终止任务

接口：

```text
POST /kill
```

请求：

```json
{
  "job_id": 1001,
  "log_id": 2001
}
```

响应：

```json
{
  "code": 0,
  "message": "Kill success",
  "data": {
    "log_id": 2001,
    "status": "killing"
  }
}
```

处理规则：

1. 校验 token。
2. 根据 `log_id` 查找本地运行实例。
3. 找不到运行实例时返回明确错误。
4. 找到后向进程组发送终止信号。
5. `/kill` 返回成功只表示终止请求已接收。
6. 最终状态通过 callback 返回 admin。

## 8. 核心模块实现方案

### 8.1 ProcessManager

建议文件：

```text
internal/process/manager.go
```

职责：

1. 维护 `runningByJobID`。
2. 维护 `runningByLogID`。
3. 接收 Run 请求并创建运行实例。
4. 接收 Kill 请求并找到运行实例。
5. 在任务结束后清理内存状态。

建议结构：

```go
type Manager struct {
    mu sync.Mutex
    runningByJobID map[int64]*RunInstance
    runningByLogID map[int64]*RunInstance
}
```

并发要求：

1. 所有 map 访问必须加锁。
2. 同一 `job_id` 同时只能有一个运行实例。
3. 不同 `job_id` 可以并行运行。
4. 任务结束后必须删除 `runningByJobID` 和 `runningByLogID`。

### 8.2 RunInstance

建议结构：

```go
type RunInstance struct {
    JobID int64
    LogID int64
    Script string
    TimeoutSeconds int32
    CallbackURL string
    CallbackToken string
    StartedAt time.Time
    Cmd *exec.Cmd
    Pgid int
    Done chan struct{}
}
```

说明：

1. `LogID` 是 callback 的幂等关键。
2. `Pgid` 用于进程组终止。
3. `Done` 用于 kill 或 timeout 等待。

### 8.3 ShellRunner

建议文件：

```text
internal/process/runner.go
```

执行流程：

```text
1. 创建临时 .sh 文件
2. 写入 script 内容
3. chmod 0700
4. exec.CommandContext(shell_path, temp_file)
5. 设置 Linux 进程组
6. 启动命令
7. 并发读取 stdout/stderr
8. 等待命令结束或 timeout
9. 收集 exit_code
10. 删除临时 .sh 文件
11. 生成 callback result
12. 写 pending 文件
13. 触发 callback
14. 清理运行实例
```

临时文件规则：

1. 路径在 `executor.temp_dir` 下。
2. 文件名包含 `job_id` 和 `log_id`。
3. 权限使用 `0700`。
4. 执行结束后删除。

### 8.4 Linux 进程组

建议文件：

```text
internal/process/killer_linux.go
```

启动脚本时必须设置独立进程组。

Go 实现方向：

```go
cmd.SysProcAttr = &syscall.SysProcAttr{
    Setpgid: true,
}
```

终止时：

```text
1. syscall.Kill(-pgid, syscall.SIGTERM)
2. 等待 kill_grace_seconds，默认 5 秒
3. 如果仍未退出，syscall.Kill(-pgid, syscall.SIGKILL)
```

注意：

1. 负数 pid 表示进程组。
2. 该能力仅支持 Linux。
3. V1 不保证清理 `nohup`、`setsid`、daemon 化后脱离进程组的进程。
4. 如果 Python 脚本需要释放资源，应自行处理 `SIGTERM`。

### 8.5 日志收集

建议文件：

```text
internal/process/log_buffer.go
```

要求：

1. 合并 stdout 和 stderr。
2. 单次日志最大 5MB。
3. 超过后设置 `log_truncated = true`。
4. 推荐保留前 2.5MB 和后 2.5MB。
5. 中间插入截断提示文本。

不要直接无限 `CombinedOutput()`。

原因：

1. 长任务输出可能很大。
2. `CombinedOutput()` 会把所有日志留在内存中。
3. 需要明确日志截断策略。

### 8.6 PendingStore

建议文件：

```text
internal/store/pending_store.go
```

目录：

```text
/app/data/callbacks/pending/{log_id}.json
/app/data/callbacks/expired/{log_id}.json
```

能力：

1. `Save(result CallbackResult) error`
2. `ListPending() ([]CallbackResult, error)`
3. `Delete(logID int64) error`
4. `Expire(logID int64) error`

写文件要求：

1. 使用 JSON。
2. 先写临时文件，再原子 rename。
3. 文件名使用 `log_id`。
4. 执行器重启后可以扫描 pending 目录恢复回调。

### 8.7 CallbackClient

建议文件：

```text
internal/callback/client.go
```

请求：

```text
POST callback_url
X-Callback-Token: callback_token
```

成功条件：

1. HTTP 状态码为 2xx。
2. 响应 `code = 0`。

失败处理：

1. 记录本地日志。
2. 不删除 pending 文件。
3. 等待 worker 重试。

### 8.8 CallbackWorker

建议文件：

```text
internal/callback/worker.go
```

规则：

1. 服务启动后立即扫描 pending 目录。
2. 每 30 秒扫描一次。
3. 对 pending 文件逐个 callback。
4. callback 成功后删除 pending 文件。
5. pending 文件超过 7 天仍失败，移动到 expired 或删除。
6. 记录本地日志。

## 9. 状态结果生成

执行器 callback 给 admin 的状态必须符合 PRD。

### 9.1 success

条件：

1. 脚本自然退出。
2. exit_code = 0。

### 9.2 failed

条件：

1. 脚本自然退出。
2. exit_code 非 0。
3. 启动脚本失败。
4. 创建临时文件失败。
5. 执行器内部执行异常。

### 9.3 timeout

条件：

1. 超过 `timeout_seconds`。
2. 执行器主动终止进程组。

### 9.4 killed

条件：

1. admin 调用 `/kill`。
2. 执行器完成进程组终止。

### 9.5 状态优先级

如果 timeout 和 kill 同时发生，建议优先以先到达的控制事件为准：

1. 已经因 timeout 进入终止流程，则 callback `timeout`。
2. 已经因用户 kill 进入终止流程，则 callback `killed`。
3. 实现中需要通过运行实例内的状态字段加锁保护。

## 10. 错误码规划

在 `internal/errors/codes.go` 中追加执行器业务错误。

建议错误：

```text
ErrExecutorTokenMissing
ErrExecutorTokenInvalid
ErrRunInvalidRequest
ErrRunJobAlreadyRunning
ErrRunScriptEmpty
ErrRunCreateTempFileFailed
ErrRunStartProcessFailed
ErrKillLogNotRunning
ErrKillProcessFailed
ErrPendingWriteFailed
ErrPendingReadFailed
ErrCallbackFailed
ErrCallbackConfigInvalid
ErrConfigInvalid
```

要求：

1. 不暴露本地文件绝对路径中的敏感信息。
2. 不把系统底层错误原样返回给 admin。
3. 本地日志中可以记录详细错误。

## 11. Server 注册方案

`internal/server/http.go` 需要注册：

```text
GET  /health
POST /run
POST /kill
```

这些是执行器内部接口，路径以 PRD 为准。

建议：

1. `/health` 可以手写路由。
2. `/run` 和 `/kill` 可以 proto-first，也可以手写 HTTP。
3. 如果手写 HTTP，仍必须复用统一错误响应，不得写临时 JSON 错误结构。
4. HTTP 层不写业务逻辑，只调用 `service`。

## 12. 依赖注入要求

需要调整 `ProviderSet`：

1. 移除 `NewUserService` 相关示例。
2. 移除 `NewUserUsecase` 相关示例。
3. 移除 `NewUserRepo`。
4. 移除 `NewDB`。
5. 移除 `NewTransaction`，除非有非数据库事务抽象需求。
6. 新增 `ProcessManager`、`PendingStore`、`CallbackClient`、`CallbackWorker`、`ExecutorService` 等 provider。

修改后运行：

```bash
make config
make wire
make test
```

## 13. Docker 部署注意事项

执行器可以跑在 Docker 容器内。

必须挂载：

```bash
docker run \
  -v /data/scripts:/data/scripts \
  -v /data/chronoflow-exec:/app/data \
  chronoFlow-exec
```

规则：

1. Glue Shell 中使用容器内路径。
2. 业务 Python 脚本和 venv 通过 volume 挂载进容器。
3. pending callback 文件保存在 `/app/data` 下。
4. 临时 `.sh` 文件保存在 `/app/data/tmp` 下。
5. 执行器只终止容器内由本次任务启动的进程组。

## 14. 测试方案

### 14.1 Service 层

重点测试：

1. 缺 token 返回认证错误。
2. token 错误返回认证错误。
3. `/run` 缺 script 返回参数错误。
4. `/run` 缺 callback_url 返回参数错误。
5. `/kill` 缺 log_id 返回参数错误。
6. 成功响应结构为 `code/message/data`。

### 14.2 ProcessManager

重点测试：

1. 同一 job_id 第二次 run 被拒绝。
2. 不同 job_id 可以并行。
3. 任务结束后 running map 被清理。
4. kill 可以根据 log_id 找到运行实例。
5. 找不到 log_id 返回错误。

### 14.3 Runner

重点测试：

1. 简单 echo 脚本成功。
2. exit 1 脚本返回 failed。
3. sleep 超过 timeout 返回 timeout。
4. stdout/stderr 被合并。
5. 日志超过上限被截断。
6. 临时脚本文件最终被删除。

### 14.4 Killer

重点测试：

1. kill running 进程后返回 killed。
2. Python 子进程或 shell 子进程能随进程组终止。
3. SIGTERM 后未退出时发送 SIGKILL。

Linux 进程组测试需要在 Linux 或 Linux 容器中执行。

### 14.5 PendingStore

重点测试：

1. 保存 pending JSON。
2. 读取 pending 列表。
3. callback 成功后删除 pending。
4. pending 过期后移动到 expired。
5. JSON 损坏时记录错误并跳过。

### 14.6 CallbackWorker

重点测试：

1. callback 成功删除文件。
2. callback 失败保留文件。
3. 重启后可以重新扫描 pending。
4. 超过 7 天过期。

## 16. AI 实现禁令

1. 不要让执行器连接 MySQL。
2. 不要保留 GORM 初始化作为执行器启动依赖。
3. 不要为执行器创建数据库表。
4. 不要把模板 `user` 示例当业务代码。
5. 不要在执行器中查询或修改 admin 的数据库。
6. 不要在执行器中解析 Cron。
7. 不要在执行器中保存最终日志正文。
8. 不要无限制收集 stdout/stderr。
9. 不要只 kill 主进程而忽略进程组。
10. 不要把 callback 失败的结果只放内存里。
11. 不要让 `/run` 等待脚本执行完成。
12. 不要把 Docker 容器内路径和宿主机路径混为一谈。
