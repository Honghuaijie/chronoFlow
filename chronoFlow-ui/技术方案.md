# ChronoFlow UI 技术方案

## 1. 文档定位

本文档用于指导 AI 或开发者在 `chronoFlow-ui` 中实现 ChronoFlow V1 调度中心前端。

本文档是开发执行方案，不是概念介绍。实现时以以下文件为需求来源：

1. `../docs/prd-v1.md`
2. 当前项目的 `FRONTEND_AI_CODING_RULES.md`
3. `chronoFlow-admin/技术方案.md`

`chronoFlow-ui` 是面向内网单团队的后台操作台，不是官网、落地页或营销页面。首屏应直接进入登录页或调度中心工作台，不做宣传型 hero。

本方案已按 `ui-ux-pro-max` 的后台工具设计建议补充：ChronoFlow UI 应采用信息密集、状态清晰、表格优先的运维控制台风格，而不是宣传型、卡片堆叠型或装饰型界面。

## 2. 技术栈

固定使用：

1. Vue3
2. TypeScript
3. Ant Design Vue
4. Pinia
5. Vue Router
6. Axios 或项目内统一 request 封装

前端固定调用链：

```text
view/page -> store -> api
```

禁止：

1. 页面直接请求接口。
2. 页面直接处理 HTTP Response。
3. 页面直接构造 FormData。
4. 页面中滥用 `any`。
5. Store 拼 UI 结构。
6. API 层管理页面状态。

## 3. UI/UX 设计系统

### 3.1 产品界面定位

ChronoFlow 是定时任务运维控制台，用户高频操作是查看状态、筛选任务、启动停止任务、查看日志和处理失败。

界面设计目标：

1. 信息密集但不拥挤。
2. 表格可扫描。
3. 状态反馈明确。
4. 危险操作有确认。
5. 长任务状态可持续感知。
6. 日志阅读舒适。
7. 移动端可用，但桌面端优先。

不要做：

1. 营销型首屏。
2. 大面积装饰背景。
3. 大卡片堆叠式后台。
4. 过多渐变和装饰性视觉。
5. 深色大面积蓝色背景。

### 3.2 视觉风格

整体风格：

```text
安静、专业、运维工具、数据密集、低装饰。
```

推荐 Ant Design Vue token：

```ts
const themeTokens = {
  colorPrimary: '#1E40AF',
  colorInfo: '#3B82F6',
  colorSuccess: '#16A34A',
  colorWarning: '#D97706',
  colorError: '#DC2626',
  colorBgLayout: '#F8FAFC',
  colorBorder: '#D9E2F2',
  borderRadius: 6,
  fontSize: 14,
}
```

使用要求：

1. 主色只用于主按钮、选中态、链接和焦点态。
2. 告警和终止相关操作使用 amber / red，不要用主蓝色表达危险。
3. 背景以浅灰白为主，不使用大面积深蓝或纯黑。
4. 卡片圆角不超过 8px。
5. 页面区块不要做成层层嵌套卡片。

### 3.3 字体与代码显示

普通界面文字使用系统字体：

```css
font-family:
  -apple-system,
  BlinkMacSystemFont,
  "Segoe UI",
  "PingFang SC",
  "Microsoft YaHei",
  sans-serif;
```

日志、Cron、Shell、路径、token 脱敏值使用等宽字体：

```css
font-family:
  "Fira Code",
  "SFMono-Regular",
  Consolas,
  "Liberation Mono",
  monospace;
```

要求：

1. 中文界面不要强制使用不适合中文的英文字体。
2. 日志查看器必须使用等宽字体。
3. 不使用随视口宽度变化的字体大小。
4. 字间距保持默认，不使用负字距。

### 3.4 布局密度

桌面端优先视口：

```text
1024px / 1280px / 1440px / 1920px
```

基础布局：

1. 左侧导航宽度 208-224px。
2. 顶部栏高度 48-56px。
3. 内容区左右 padding 24px。
4. 表格页顶部工具栏固定为一行或两行。
5. 筛选项过多时使用折叠筛选，不把页面顶部撑太高。

移动端：

1. 375px 宽度下表格允许横向滚动。
2. 操作按钮可折叠进下拉菜单。
3. 不要求移动端拥有最佳效率，但不能布局破碎。

### 3.5 交互反馈

必须实现：

1. 请求超过 300ms 显示 loading 或 skeleton。
2. 表单提交按钮进入 loading 后禁用。
3. 删除、停止、终止等危险操作必须二次确认。
4. 表单字段优先在 blur 时校验。
5. 操作成功和失败都必须有反馈。
6. 轮询中状态变化要自然刷新，不要闪烁整张表。
7. 可点击元素必须有 hover、active、focus 状态。

动效：

1. 只使用 150-300ms 的轻量过渡。
2. 不使用大幅度动画。
3. 尊重 `prefers-reduced-motion`。

### 3.6 可访问性

最低要求：

1. 文本对比度满足 WCAG AA。
2. 键盘焦点态清晰可见。
3. 图标按钮必须有 tooltip 或 aria-label。
4. 表单错误要贴近字段显示。
5. 状态不要只靠颜色区分，必须有文字。
6. 所有危险确认弹窗默认焦点不能落在危险按钮上。

## 6. API 封装规则

### 6.1 统一响应类型

后端响应统一为：

```ts
export interface ApiResponse<T> {
  code: number
  message: string
  data: T
}
```

API 层返回给 Store 的应是解包后的 `data`。

示例：

```ts
export async function listExecutors(params: ListExecutorsParams): Promise<ListExecutorsData> {
  const res = await request.get<ApiResponse<ListExecutorsData>>('/v1/admin/executors/list', { params })
  return unwrap(res)
}
```

### 6.2 Request 拦截器

`src/api/request.ts` 负责：

1. 设置 baseURL。
2. 注入登录 token。
3. 统一处理 `code != 0`。
4. 处理 401 跳转登录。
5. 抛出业务异常给 Store。

不要在页面里判断原始 HTTP 状态。

### 6.3 接口模块

Auth：

```text
POST /v1/public/auth/login
POST /v1/admin/auth/logout
GET  /v1/admin/auth/current
POST /v1/admin/auth/changePassword
```

Executors：

```text

POST /v1/admin/executors/create
POST /v1/admin/executors/update
POST /v1/admin/executors/delete
POST /v1/admin/executors/check
```

Jobs：


```text

```

## 7. 类型设计

### 7.1 执行器类型

```ts
export type ExecutorStatus = 'online' | 'offline'

export interface ExecutorItem {
  id: number
  name: string
  address: string
  token_masked: string
  description: string
  status: ExecutorStatus
  last_heartbeat_time: string
  created_at: string
  updated_at: string
}
```

注意：

1. 前端永远不接收完整 executor token。
2. 编辑执行器时 token 输入框为空表示不修改 token。

### 7.2 任务类型

```ts
export type ScheduleStatus = 'stopped' | 'running'
export type LastRunStatus =
  | 'none'
  | 'running'
  | 'killing'
  | 'success'
  | 'failed'
  | 'timeout'
  | 'skipped'
  | 'killed'

export interface JobItem {
  id: number
  executor_id: number
  name: string
  cron_expr: string
  timeout_seconds: number
  schedule_status: ScheduleStatus
  last_run_status: LastRunStatus
  description: string
  created_at: string
  updated_at: string
}
```

### 7.3 日志类型

```ts
export type TriggerType = 'manual' | 'cron'

  | 'running'

export interface JobLogItem {
```

```ts
export interface JobLogDetail extends JobLogItem {
```

## 8. Store 设计

### 8.1 AuthStore

职责：

1. 保存 token。
2. 保存当前管理员信息。
3. 登录。
4. 退出。
5. 获取当前用户。
6. 修改密码。

状态：

```ts
token: string
currentUser: CurrentUser | null
loginLoading: boolean
currentLoading: boolean
```

### 8.2 ExecutorStore

职责：

1. 执行器列表。
2. 新增执行器。
3. 编辑执行器。
4. 删除执行器。
5. 手动健康检查。

状态：

```ts
items: ExecutorItem[]
total: number
loading: boolean
submitLoading: boolean
filters: ExecutorFilters
```

### 8.3 JobStore

职责：

1. 按执行器查询任务。
2. 新增任务。
3. 编辑任务。
4. 删除任务。
5. 启动任务。
6. 停止任务。
7. 手动运行任务。
8. 终止任务。

状态：

```ts
selectedExecutorId: number | null
items: JobItem[]
total: number
loading: boolean
submitLoading: boolean
actionLoadingById: Record<number, boolean>
```

要求：

1. 未选择执行器时不请求任务列表。
2. 手动运行成功后提示“任务已下发”，不要提示“执行成功”。
3. 任务状态为 `running` 或 `killing` 时，手动运行按钮置灰。

### 8.4 GlueStore

职责：

1. 获取任务 Glue。
2. 保存任务 Glue。

状态：

```ts
jobId: number | null
content: string
loading: boolean
saving: boolean
```

### 8.5 JobLogStore

职责：

1. 日志列表。
2. 日志详情。
3. 状态筛选。
4. 触发方式筛选。
5. 时间范围筛选。
6. 终止 running 日志。

状态：

```ts
items: JobLogItem[]
total: number
loading: boolean
detail: JobLogDetail | null
detailLoading: boolean
filters: JobLogFilters
```

表格体验：

### 9.4 任务管理页

路径：

```text
/jobs
```

功能：

1. 先选择执行器。
2. 查看该执行器下任务列表。
3. 新增任务。
4. 编辑任务。
5. 编辑 Glue。
6. 手动运行。
7. 启动。
8. 停止。
9. 终止。
10. 删除。

页面结构：

```text
顶部：执行器选择器
中部：任务表格
右侧或弹窗：任务表单 / Glue 编辑
```

任务管理页是高频页面，桌面端建议使用左右分区：

```text
左侧：执行器选择 + 任务筛选
右侧：任务表格和操作
```

如果实现复杂，V1 可以先使用顶部执行器选择器。

任务表格列：

```text
任务名称
Cron
超时时间
调度状态
最近执行状态
描述
更新时间
操作
```

按钮规则：

1. 无 Glue：启动、手动运行置灰。
2. `schedule_status = stopped`：显示启动。
3. `schedule_status = running`：显示停止。
4. `last_run_status = running`：显示终止，手动运行置灰。
5. `last_run_status = killing`：显示终止中，手动运行置灰。
6. 删除前二次确认。

手动运行提示：

```text
任务已下发，请到执行日志查看运行结果。
```

不要提示：

```text
任务执行成功。
```

因为 `/run` 是异步 accepted，不代表脚本成功。

交互要求：

1. 启动、停止、运行、终止按钮使用图标 + tooltip 或图标 + 短文字。
2. 终止和删除必须使用危险确认。
3. 手动运行成功后可以提供“查看日志”快捷入口。
4. running/killing 行可以轻微高亮，但不要使用强烈背景色。

### 9.5 Glue 编辑

建议使用抽屉或弹窗。

功能：

1. 编辑多行 Shell。
2. 保存。
3. 关闭前如果有未保存内容，提示确认。

编辑器：

1. V1 可以先用 `a-textarea`。
2. 后续可以替换为 Monaco Editor。
3. 文案和标题使用“Glue Shell”。
4. Shell 内容使用等宽字体。
5. 高度建议不少于 420px。
6. 保存按钮固定在抽屉底部，避免长脚本滚动后找不到操作。

默认示例可以作为 placeholder：

```bash
#!/bin/bash

cd /data/scripts/data-process
source venv/bin/activate

python3 sync_card_status.py
```

### 9.7 日志详情页

路径：

```text

```

2. 执行状态。

7. Glue 快照。
8. 日志正文。

日志查看器：

1. 使用等宽字体。

3. 支持换行开关。
4. 日志文件不存在时显示“日志文件不存在或已被清理”。

7. 日志容器高度应稳定，不因内容加载导致页面大幅跳动。
8. 长日志使用虚拟滚动或固定高度滚动容器，避免页面卡顿。

终止按钮：

1. 仅 `status = running` 显示。
2. 点击后二次确认。
3. 成功后状态可立即刷新为 `killing`。

2. 修改密码。
3. 查看系统配置摘要。

配置摘要可以展示：

1. Cron 时区。
2. 日志保留天数。
3. 单次日志上限。
4. callback pending 保留策略说明。

V1 不要求支持在线修改这些配置。

## 10. 状态展示规范

### 10.1 执行器状态

```text
online  -> 绿色：在线
offline -> 红色：离线
```

### 10.2 调度状态

```text
running -> 绿色：已启动
stopped -> 默认色：已停止
```

### 10.3 执行状态

```text
running -> 蓝色：执行中
killing -> 橙色：终止中
success -> 绿色：成功
failed  -> 红色：失败
timeout -> 红色：超时
skipped -> 默认色：跳过
killed  -> 橙色：已终止
none    -> 默认色：未执行
```

建议封装：

```text
components/StatusTag.vue
utils/status.ts
```

页面不要散落状态文案和颜色。

## 11. 表单校验规则

### 11.1 执行器表单

1. 名称必填。
2. 地址必填，必须以 `http://` 或 `https://` 开头。
3. 新增时 token 必填。
4. 编辑时 token 可空。
5. 描述最多 500 字。
6. 地址输入框使用 URL 类型或自定义 URL 校验。
7. token 输入框使用 password 类型，支持显示/隐藏。

### 11.2 任务表单

1. 任务名称必填。
2. Cron 必填。
3. Cron 前端做基础非空校验，最终以后端校验为准。
4. 超时时间默认 600 秒。
5. 超时时间必须大于 0。
6. 描述最多 500 字。
7. Cron 输入框旁提供最近一次解析说明或示例提示。

### 11.3 Glue 表单

1. 内容必填。
2. 允许多行 Shell。
3. 保存时 trim 后不能为空。

### 11.4 修改密码表单

1. 原密码必填。
2. 新密码必填。
3. 确认密码必须与新密码一致。

## 12. 错误处理

API 层：

1. 解包后端 `code/message/data`。
2. `code != 0` 抛出业务错误。
3. 401 清理 token 并跳转登录。

Store 层：

1. 捕获业务错误。
2. 控制 loading 状态恢复。
3. 决定是否弹出 message。

页面层：

1. 不解析原始错误结构。
2. 根据 Store 状态展示 loading 和错误提示。

典型提示：

1. 手动运行 accepted：`任务已下发，请到执行日志查看运行结果`。
2. 手动运行冲突：展示后端返回的 `任务正在执行中`。
3. callback 和执行器内部错误不在 UI 直接处理，只通过日志状态展示。

## 13. 轮询策略

V1 不做 WebSocket。

需要轮询的地方：

1. 任务列表中存在 running/killing 状态时。
2. 日志列表中存在 running/killing 状态时。
3. 日志详情状态为 running/killing 时。

建议：

1. 轮询间隔 5 秒。
2. 页面不可见或离开路由时停止。
3. 同一个 Store 内避免重复定时器。
4. 请求未完成时不发起下一次同类轮询。
5. 页面右上角可以显示轻量的“自动刷新中”状态。

## 14. 与后端契约的关键理解

1. `/run` 是异步 accepted，不代表执行成功。
2. 日志最终状态以执行器 callback 后 admin 返回的数据为准。
3. 日志正文来自 admin 文件系统，不来自 MySQL。
4. `log_truncated = true` 表示日志被截断。
5. `killing` 是中间态，最终会变成 `killed` 或 `failed`。
6. 手动运行冲突不生成日志。
7. Cron 冲突生成 skipped 日志。
8. 删除任务不会立即删除历史日志。
9. 执行器 token 不回显完整值。

## 16. 测试与自检

### 16.1 类型检查

必须保证：

1. 无隐式 any。
2. API 请求和响应都有类型。
3. Store state 有类型。
4. 表单模型有类型。

### 16.2 关键场景

至少手动验证：

1. 未登录访问后台跳转登录。
2. 登录成功进入执行器管理。
3. 新增执行器时 token 必填。
4. 编辑执行器时 token 可空。
5. 未选择执行器时任务列表不请求。
6. 无 Glue 任务不能启动和手动运行。
7. 手动运行后提示“已下发”。
8. running 任务手动运行按钮置灰。
9. running 日志可以终止。
10. killing 状态展示为终止中。
11. 日志详情能展示 Glue 快照和日志正文。
12. 日志截断时有提示。
13. 日志文件不存在时显示明确提示文案。
14. 375px 视口下表格不撑破页面。
15. 键盘 Tab 可以看到清晰焦点态。
16. 请求 loading 时按钮不可重复提交。

### 16.3 建议命令

根据最终前端工程选择对应命令，至少应包含：

```bash
npm run typecheck
npm run lint
npm run build
```

如果项目使用 pnpm，则使用：

```bash
pnpm typecheck
pnpm lint
pnpm build
```

## 17. UI 交付检查清单

1. 桌面端表格列没有明显重叠。
2. 移动端表格有横向滚动或替代卡片布局。
3. 所有表单提交都有 loading。
4. 所有危险操作都有二次确认。
5. 所有状态都有颜色和文字。
6. 所有图标按钮都有 tooltip 或 aria-label。
7. 日志正文使用等宽字体。

9. 页面在 375px、768px、1024px、1440px 下可用。
10. `prefers-reduced-motion` 下无强制动画。

## 18. AI 实现禁令

1. 不要做营销页或 landing page。
2. 不要让页面直接请求接口。
3. 不要让页面直接处理原始 HTTP Response。
4. 不要滥用 `any`。
5. 不要把 `/run` accepted 展示为“执行成功”。
6. 不要在前端保存或展示完整 executor token。
7. 不要用前端按钮隐藏代替后端权限控制。
8. 不要在多个页面重复写状态颜色和文案。
9. 不要把日志详情的大段文本放进表格列表。
10. 不要引入 WebSocket，V1 使用轮询。
11. 不要在系统设置页做 V1 未支持的在线配置修改。
12. 不要把 Docker、执行器部署说明写成前端功能。
13. 不要用颜色作为唯一状态表达。
14. 不要让宽表格在移动端撑破视口。
15. 不要使用 emoji 作为正式图标。
16. 不要使用大面积深色背景或单一蓝色系铺满整个后台。
