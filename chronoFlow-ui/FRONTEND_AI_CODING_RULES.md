# AI 前端代码生成规范

## 适用范围

这份文档用于约束 AI 在当前模板体系内生成前端代码时的行为，目标是生成分层清晰、类型明确、可维护的前端代码。

适用范围：

- 前端：`Vue3 + Ant Design Vue + Pinia + TypeScript`
- 使用场景：新增页面、列表页、表单页、详情页、上传页

不适用范围：

- 历史脏代码迁移
- 跨技术栈重构
- 脱离当前模板目录结构的自由发挥


## 前端编码规范

### 技术栈与分层

前端固定使用：

- `Vue3`
- `Ant Design Vue`
- `Pinia`
- `TypeScript`

固定调用链：

- `page/view -> store -> api`

### 目录职责

推荐目录职责如下：

- `src/api/`：接口请求函数、请求参数类型、响应类型
- `src/stores/`：页面状态、分页状态、提交状态、请求编排
- `src/views/`：页面容器，负责布局和触发交互
- `src/components/`：可复用业务组件和通用组件
- `src/types/`：前端视图模型和通用类型
- `src/router/`：路由定义

### 页面、Store、API 规则

页面必须遵守：

- 页面不得直接请求接口
- 页面不得直接构造 `FormData` 并提交
- 页面只负责收集输入、触发动作、展示状态

`store` 必须遵守：

- 统一管理 `loading`
- 统一管理分页状态
- 统一管理列表数据和表单提交状态
- 统一编排调用 `api`

`api` 必须遵守：

- 只封装请求
- 不处理页面状态
- 文件上传场景由 `api` 层负责构造 `FormData`
- 返回给 `store` 的应当是解包后的业务数据，而不是原始 HTTP Response 对象

错误处理规则：

- 全局拦截器或 `api` 层负责处理统一响应和错误解包
- `api` 层向 `store` 抛出业务异常
- `store` 负责决定页面级提示、重试和状态恢复

### 类型约束

必须执行全链路 TypeScript：

- API 请求参数必须有类型
- API 响应必须有类型
- `store state` 必须有类型
- 表单模型必须有类型
- 组件 `props` / `emits` 必须有类型

禁止：

- 滥用 `any`
- 滥用 `Record<string, any>`
- 依赖页面内的隐式字段推导代替明确类型

### 文件上传前端规则

文件上传场景必须遵守：

- 上传组件通过 `store` 发起上传
- `api` 层负责构造 `FormData`
- 页面只负责收集文件、触发上传、展示结果
- 前端可以做文件大小和类型预校验
- 后端 `service` 必须做最终兜底校验

## AI 禁止生成清单

### 前端禁止项

- 页面直接发起请求
- 页面直接构造 `FormData`
- 组件里散落全局状态管理
- `store` 拼 UI 结构
- 用 `any` 接上传响应或列表响应
