# ChronoFlow Admin 技术方案

## 1. 文档定位

本文档用于指导 AI 或开发者在 `chronoFlow-admin` 中实现 ChronoFlow V1 调度器后端。

本文档是开发执行方案，不是概念介绍。实现时以以下文件为需求来源：

1. `../prd-v1.md`
2. 当前项目的 `AI_CODING_RULES.md`
3. 当前项目的 `README.md`
4. 当前项目的 `TESTING_GUIDE.md`

`chronoFlow-admin` 是平台唯一连接 MySQL 的服务，负责用户登录、执行器管理、任务管理、Glue 管理、Cron 调度、执行日志元数据、日志正文文件、执行器健康检查、任务下发、执行结果 callback、调度器重启恢复和日志清理。

## 2. 模板使用原则

当前后端模板中已有 `user` 示例模块，包括：

1. `api/user/v1/user.proto`
2. `internal/service/user.go`
3. `internal/biz/user.go`
4. `internal/data/user.go`
5. `internal/data/model.go`
6. `internal/server/http.go` 中的示例路由

这些示例只用于学习模板写法，不是 ChronoFlow 业务代码。

实现 ChronoFlow 时必须遵守：

1. 可以参考 `user` 示例的 proto、service、biz、data、错误处理和测试写法。
2. 不要把 `user` 示例接口当作真实业务接口。
3. 不要围绕示例 `User` 扩展 ChronoFlow 业务。
4. ChronoFlow 业务模块应围绕 `auth`、`executors`、`jobs`、`glues`、`job_logs`、`scheduler`、`callbacks` 等领域重新建立。
5. 如果示例模块影响业务结构，可以在正式实现阶段删除或隔离。

## 3. 架构边界

### 3.1 Admin 负责

1. 连接 MySQL。
2. 管理所有数据库表。
3. 保存执行日志正文文件。
4. 对前端提供管理接口。
5. 通过 HTTP 调用执行器。
6. 接收执行器 callback。
7. 维护 Cron 调度器。
8. 做执行器健康检查。
9. 做任务互斥判断。
10. 做调度器重启恢复。
11. 做日志保留清理。

### 3.2 Admin 不负责

1. 直接执行 Shell 脚本。
2. 直接管理 Linux 进程组。
3. 保存执行器 pending callback 文件。
4. 同步业务 Python 脚本。
5. 创建 Python 虚拟环境。

### 3.3 与执行器的关系

Admin 调用执行器：

```text
GET  /health
POST /run
POST /kill
```

执行器回调 Admin：

```text
POST /internal/job-runs/callback
```

Admin 调用执行器时使用执行器自己的 `X-Executor-Token`。

执行器回调 Admin 时使用全局 `X-Callback-Token`。

## 5. 配置设计

需要扩展 `internal/conf/conf.proto` 和 `configs/config*.yaml`。

建议配置结构：

```yaml
server:
  http:
    network: tcp
    addr: 0.0.0.0:10003
    timeout: 30s
  grpc:
    network: tcp
    addr: 0.0.0.0:11003
    timeout: 30s
  public_base_url: "http://127.0.0.1:10003"

data:
  database:
    driver: mysql
    host: "${DB_HOST:127.0.0.1}"
    port: "${DB_PORT:3306}"
    username: "${DB_USER:root}"
    password: "${DB_PASSWORD:root}"
    database: "${DB_NAME:chronoflow}"
    charset: utf8mb4
    parseTime: true
    loc: Local

scheduler:
  timezone: "Asia/Shanghai"

executor:
  health_check_interval_seconds: 10
  health_check_fail_threshold: 3
  request_timeout_seconds: 10

security:
  jwt_secret: "${JWT_SECRET:change-me}"
  token_encrypt_key: "${CHRONOFLOW_TOKEN_ENCRYPT_KEY:}"
  callback_token: "${CHRONOFLOW_CALLBACK_TOKEN:}"

logs:
  data_dir: "/app/data"
  max_log_bytes: 5242880
  retention_days: 30
  cleanup_cron: "0 0 3 * * *"

recovery:
  startup_running_grace_seconds: 120
  killing_timeout_seconds: 60
```

实现要求：

1. `token_encrypt_key` 必须是 32 字节密钥。
2. 生产环境不允许空 `token_encrypt_key`。
3. 生产环境不允许空 `callback_token`。
4. 环境变量优先覆盖配置文件。
5. `public_base_url` 用于拼出执行器回调地址。

修改配置后必须运行：

```bash
make config
make wire
```

## 7. 状态枚举

建议在 `internal/biz` 或单独 `internal/domain` 中定义字符串常量。

```go
const (
    ScheduleStatusStopped = "stopped"
    ScheduleStatusRunning = "running"

    ExecutorStatusOnline  = "online"
    ExecutorStatusOffline = "offline"

    TriggerTypeManual = "manual"
    TriggerTypeCron   = "cron"

    JobLogStatusRunning = "running"
    JobLogStatusKilling = "killing"
    JobLogStatusSuccess = "success"
    JobLogStatusFailed  = "failed"
    JobLogStatusTimeout = "timeout"
    JobLogStatusSkipped = "skipped"
    JobLogStatusKilled  = "killed"
)
```

状态流转要求：

1. `running -> success / failed / timeout / killed`
2. `running -> killing`
3. `killing -> killed / failed / timeout`
4. `skipped` 创建后就是最终状态。
5. 最终状态不允许被 callback 再次覆盖。

## 8. 接口设计

### 8.1 接口风格

遵守模板方法级路径风格：

```text
GET  /v1/admin/executors/list
POST /v1/admin/executors/create
POST /v1/admin/executors/update
POST /v1/admin/executors/delete
```

不要写 REST 风格：

```text
GET /v1/executors
PUT /v1/executors/{id}
DELETE /v1/executors/{id}
```

### 8.2 Auth 接口

登录接口不需要管理员登录态，但属于后台登录入口。建议路径：

```text
POST /v1/public/auth/login
POST /v1/admin/auth/logout
GET  /v1/admin/auth/current
POST /v1/admin/auth/changePassword
```

说明：

1. `login` 校验账号密码，返回 JWT。
2. `logout` V1 可前端删除 token，后端返回成功。
3. `current` 返回当前管理员信息。
4. `changePassword` 修改当前管理员密码。

### 8.3 Executor 接口

```text
GET  /v1/admin/executors/list
POST /v1/admin/executors/create
POST /v1/admin/executors/update
POST /v1/admin/executors/delete
POST /v1/admin/executors/check
```

关键规则：

1. 创建和更新时 token 入库前必须加密。
2. 列表和详情不得返回完整 token。
3. 删除执行器前必须检查是否存在未删除任务。
4. 手动检测时调用执行器 `/health`。

### 8.4 Job 接口

```text
GET  /v1/admin/jobs/list
POST /v1/admin/jobs/create
POST /v1/admin/jobs/update
POST /v1/admin/jobs/delete
POST /v1/admin/jobs/start
POST /v1/admin/jobs/stop
POST /v1/admin/jobs/run
POST /v1/admin/jobs/kill
```

关键规则：

1. `list` 必须按 `executor_id` 筛选。
2. `create/update` 必须校验 6 位 Cron。
3. 新建任务默认 `schedule_status = stopped`。
4. `start` 前必须检查 Glue 存在。
5. `start` 后要注册 Cron。
6. `stop` 后要移除 Cron。
7. `run` 是异步下发，成功只代表执行器 accepted。
8. `kill` 将当前 running 日志改为 killing，再调用执行器 `/kill`。

### 8.5 Glue 接口

```text
GET  /v1/admin/glues/get
POST /v1/admin/glues/save
```

关键规则：

1. 每个任务最多一条 Glue。
2. 保存 Glue 时做非空校验。
3. 当前正在执行的任务不受 Glue 编辑影响。
4. 执行日志使用执行开始时的 Glue 快照。

### 8.6 JobLog 接口

```text
GET /v1/admin/jobLogs/list
GET /v1/admin/jobLogs/detail
```

列表返回元数据，不返回完整日志正文。

详情返回：

1. 元数据。
2. Glue 快照。
3. 日志正文。
4. 是否截断。
5. 如果日志文件不存在，返回“日志文件不存在或已被清理”。

### 8.7 Internal Callback 接口

```text
POST /internal/job-runs/callback
```

Header：

```text
X-Callback-Token: <callback_token>
```

请求体：

```json
{
  "log_id": 2001,
  "job_id": 1001,
  "status": "success",
  "exit_code": 0,
  "log_content": "hello\n",
  "log_truncated": false,
  "start_time": "2026-06-10 10:00:00",
  "end_time": "2026-06-10 10:00:01",
  "duration_ms": 1000,
  "error_message": ""
}
```

处理要求：

1. 校验 `X-Callback-Token`。
2. 查询 `job_logs`。
3. 只有 `running` 或 `killing` 可以更新。
4. 如果已经是最终状态，忽略重复 callback 并返回成功。
5. 写日志文件。
6. 更新 `job_logs` 元数据。

该接口是内部接口，可以使用 proto-first，也可以作为特殊 HTTP 路由手工注册；无论采用哪种方式，都必须复用统一错误响应。

## 9. 核心模块实现方案

### 9.1 Token 加密模块

新增 `internal/biz/crypto.go` 或 `internal/security/token_crypto.go`。

能力：

1. `EncryptToken(plain string) (string, error)`
2. `DecryptToken(ciphertext string) (string, error)`

要求：

1. 使用 AES-GCM。
2. nonce 每次随机生成。
3. 密文中包含 nonce 和 ciphertext。
4. 加密密钥来自配置或环境变量。
5. 前端永远不返回完整 token。

### 9.2 ExecutorClient

新增 `internal/executorclient/client.go`。

能力：

1. `Health(ctx, address, token string) error`
2. `Run(ctx, req RunRequest) error`
3. `Kill(ctx, req KillRequest) error`

要求：

1. 所有请求带 `X-Executor-Token`。
2. 请求超时由配置控制。
3. `/run` 返回 accepted 才算下发成功。
4. `/kill` 返回 killing 才算终止请求已被接收。
5. 不在 client 中写业务状态流转，只返回调用结果。

### 9.3 LogStore

新增 `internal/logstore/file_store.go`。

能力：

1. `Write(ctx, logID, jobID int64, content string) (path string, size int64, err error)`
2. `Read(ctx, path string) (string, error)`
3. `Delete(ctx, path string) error`

要求：

1. 根目录来自 `logs.data_dir`。
2. 相对路径形如 `logs/yyyy/mm/dd/job-{job_id}/log-{log_id}.log`。
3. 写文件前创建目录。
4. 路径必须做清理，禁止读取任意目录。
5. 读取文件不存在时返回可识别业务错误。

### 9.4 SchedulerManager

新增 `internal/scheduler/manager.go`。

能力：

1. 启动 robfig/cron。
2. 注册 job。
3. 移除 job。
4. 更新 job Cron。
5. 启动时恢复已启动任务。

要求：

1. 使用配置时区，默认 `Asia/Shanghai`。
2. 内存中维护 `job_id -> cron.EntryID`。
3. 修改已启动任务 Cron 后立即重载。
4. 删除或停止任务时移除 Cron。
5. Cron 触发时调用 biz 层的 `TriggerCronJob`，不要在 scheduler 里写业务判断。

### 9.5 HealthChecker

新增 `internal/worker/health_checker.go`。

规则：

1. 每 10 秒扫描未删除执行器。
2. 调用执行器 `/health`。
3. 成功：标记 online，失败次数清零。
4. 失败：失败次数加 1。
5. 连续 3 次失败：标记 offline。
6. 从 online 变 offline 时，将该执行器下 `running/killing` 日志标记 failed。
7. 错误信息：`执行器重启或失联，执行结果未知`。

### 9.6 StartupRecovery

新增 `internal/worker/startup_recovery.go`。

规则：

1. 服务启动后恢复 Cron。
2. 查询 `running/killing` 日志。
3. 等待 120 秒恢复宽限期。
4. 宽限期内 callback 正常处理。
5. 宽限期后仍是中间状态，标记 failed。
6. 错误信息：`调度器重启，执行结果未知`。

### 9.7 KillingWatcher

新增 `internal/worker/killing_watcher.go`。

规则：

1. 定期扫描 `status = killing` 的日志。
2. 如果进入 killing 超过 60 秒仍无 callback，标记 failed。
3. 错误信息：`终止任务超时，执行结果未知`。

### 9.8 LogCleaner

新增 `internal/worker/log_cleaner.go`。

规则：

1. 默认每天 03:00 执行。
2. 清理 30 天前的 `job_logs`。
3. 删除 MySQL 元数据时同步删除日志文件。
4. 文件不存在时忽略。
5. 删除任务不立即删除历史日志。

## 10. 核心业务流程

### 10.1 手动运行

```text
1. service 校验 job_id
2. biz 查询任务、Glue、执行器
3. 校验任务未删除
4. 校验 Glue 存在
5. 校验执行器 online
6. 查询是否已有 running/killing 日志
7. 如果有，返回“任务正在执行中”，不生成日志
8. 创建 running 日志和执行快照
9. 解密 executor token
10. 构造 callback_url 和 callback_token
11. 调用 executorclient.Run
12. accepted 后返回 log_id
```

### 10.2 Cron 运行

```text
1. scheduler 触发 job_id
2. biz 查询任务、Glue、执行器
3. 如果任务停止或删除，直接跳过且不写日志
4. 如果 Glue 缺失，写 failed 日志
5. 如果执行器 offline，写 failed 日志
6. 如果已有 running/killing 日志，写 skipped 日志
7. 否则创建 running 日志
8. 调用 executorclient.Run
9. 如果下发失败，running 日志更新为 failed
```

### 10.3 Callback

```text
1. service 校验 callback token
2. service 校验 log_id/status/time 字段
3. biz 查询 job_log
4. 如果 job_log 不存在，返回错误
5. 如果 job_log 是最终状态，返回成功
6. 如果 job_log 是 running/killing，继续处理
7. logstore 写日志正文文件
8. data 更新 job_logs 元数据
9. 返回成功
```

### 10.4 终止任务

```text
1. service 校验 log_id
2. biz 查询 job_log
3. 只有 running 可以终止
4. 更新状态为 killing
5. 查询执行器并解密 token
6. 调用 executorclient.Kill
7. /kill accepted 后返回成功
8. 最终状态等待 callback
```

如果 `/kill` 调用失败：

1. 保持 `killing` 状态。
2. 写入错误信息：`终止请求已发出，但执行器暂无确认`。
3. 由 KillingWatcher 在 60 秒后兜底为 failed。

## 12. 错误码规划

在 `internal/errors/codes.go` 中追加 ChronoFlow 业务错误。

建议错误：

```text
ErrExecutorNotFound
ErrExecutorHasJobs
ErrExecutorOffline
ErrExecutorTokenInvalid
ErrJobNotFound
ErrJobRunning
ErrJobGlueMissing
ErrJobCronInvalid
ErrJobLogNotFound
ErrJobLogNotRunning
ErrCallbackTokenInvalid
ErrCallbackStatusInvalid
ErrLogFileNotFound
ErrLogFileWriteFailed
ErrTokenEncryptFailed
ErrTokenDecryptFailed
ErrSchedulerRegisterFailed
```

要求：

1. 不要直接把 GORM、HTTP client、文件系统底层错误暴露给前端。
2. 可以用 `EWithMessage` 补充具体中文提示。
3. 前端可依赖稳定错误码做提示。

## 13. 测试方案

### 13.1 Service 层

重点测试：

1. 参数缺失。
2. ID 非法。
3. Cron 非法。
4. token 不返回完整明文。
5. callback token 缺失或错误。
6. 成功响应 `code = 0`、`message = 方法名 + " success"`。

### 13.2 Biz 层

使用 fake repo、fake executor client、fake log store、fake scheduler。

重点测试：

1. 手动运行任务成功创建 running 日志并调用执行器。
2. 手动运行遇到已有 running 返回 `ErrJobRunning`，不创建日志。
3. Cron 触发遇到已有 running 创建 skipped 日志。
4. 执行器 offline 时创建 failed 日志。
5. callback 对 running 日志更新成功。
6. callback 对最终状态日志幂等返回成功。
7. kill 将 running 更新为 killing 并调用执行器。
8. 执行器 offline 时 running/killing 日志被置 failed。

### 13.3 Data 层

重点测试：

1. 软删除查询过滤。
2. running/killing 查询条件。
3. callback 更新只允许中间状态。
4. 日志清理批量查询和删除。

### 13.4 LogStore

重点测试：

1. 写入日志文件并返回相对路径。
2. 读取日志文件。
3. 文件不存在返回业务错误。
4. 删除文件幂等。
5. 防止路径穿越。

### 13.5 HTTP 层

重点测试：

1. 统一错误响应。
2. 登录接口。
3. 管理接口未登录被拒绝。
4. callback token 错误被拒绝。
5. 健康检查接口可用。

运行命令：

```bash
make api
make config
make wire
make test
```
